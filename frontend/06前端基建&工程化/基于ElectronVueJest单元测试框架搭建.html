环境搭建 ​

安装依赖 ​

  • Vue-Test-Utils(Vue+Jest)

    • 情况一：初始化项目

      • 通过脚手架 vue-cli 来新建项目的时候，如果选择了 Unit Testing 单元测试且选择的是 Jest 作为测试运行器，那么在项目创建好后，就会自动配置好单元测试需要的环境，直接能用 Vue-Test-Utils 和 Jest 的 API 来写测试用例了。

    • 情况二：已有项目

      • 在项目终端运行以下语句，直接在项目中添加一个 unit-jest 插件，会自动将需要的依赖安装配置好。

        vue add @vue/unit-jest

      • 在项目根目录下找到jest.config.js文件，补充一些额外配置

        javascript
        module.exports = {
  preset: '@vue/cli-plugin-unit-jest',
  verbose: true,
  // 文件后缀
  moduleFileExtensions: ['js', 'jsx', 'json', 'vue'],
  // 文件如何转换
  transform: {
    '^.+\\.vue$': 'vue-jest',
    '.+\\.(css|styl|less|sass|scss|svg|png|jpg|ttf|woff|woff2)$': 'jest-transform-stub',
    '^.+\\.js?$': 'babel-jest',
    // '^.+\\.(ts|tsx|js|jsx)?$': './node_modules/ts-jest',
  },
  // 处理webpack的别名，比如：将@表示/src目录
  moduleNameMapper: {
    '^@/(.*)$': '<rootDir>/src/$1'
  },
  // 忽略的文件
  transformIgnorePatterns: ['/node_modules/(?!(electron|ant-design|lodash-es))'],
  // 生成快照需要的插件
  snapshotSerializers: ['jest-serializer-vue'],
  // 需要执行哪些目录下的测试用例
  testMatch: [
    '**/tests/**/*.spec.js'
  ],
  setupFilesAfterEnv: ['<rootDir>/setup-jest.js'],
}

      • 检查.eslintrc.js配置文件中是否有jest: true字段，如果没有就补上。

        js
          env: {
    jest: true
  }

  • jest-electron(Electron+Jest)

    • 在项目运行环境中，Vue app处于render process中，由于项目设置了 nodeIntegration: true，Vue app可以通过window.require获得Electron的API。但是在测试环境中代码不处于浏览器环境中而是直接处于node进程中，所以window.require是undefined的。这里需要用到插件jest-electron，这个插件可以使得jest测试运行在Electron的renderer process 或 main process中。

    • 利用npm安装

      js
      npm i --save-dev jest-electron

    • 在jest.config.js配置文件中添加以下字段

      js
      {
  "jest": {
    "runner": "jest-electron/runner",
    "testEnvironment": "jest-electron/environment"
  }
}

    • 在项目的.env文件中添加以下三行

      js
      DEBUG_MODE=1
JEST_ELECTRON_STARTUP_ARGS='--disable-dev-shm-usage'
JEST_ELECTRON_STARTUP_ARGS='--no-sandbox'

  • 运行测试指令说明

    • 完成依赖安装之后，可以在package.json中看到新增了一条运行指令test:unit": "vue-cli-service test:unit, 直接终端运行该指令执行测试。

    • 对该指令进行补充完善

      • 添加--mode test，指定测试运行在测试环境中：vue-cli-service --mode test test:unit。

      注意：test环境下，app.getPath()路径跟DEV和PROD都不一样，之后开发时需要多做一层判断

      js
      process.env.NODE_ENV === test
      • 添加--detectOpenHandles，如果测试没有正确结束，则允许 Jest 自动关闭测试会话，以便继续测试：vue-cli-service --mode test test:unit --detectOpenHandles。

高频API介绍 ​

  • Jest 常用 API

    • describe(name, fn)：描述块，讲一组功能相关的测试用例组合在一起
    • it(name, fn, timeout)：别名 test，用来放测试用例
    • afterAll(fn, timeout)：所有测试用例跑完以后执行的方法
    • beforeAll(fn, timeout)：所有测试用例执行之前执行的方法
    • afterEach(fn)：在每个测试用例执行完后执行的方法
    • beforeEach(fn)：在每个测试用例执行之前需要执行的方法

  • Vue Test Utils常用API

    • mount：挂载该组件的同时渲染其包含的子组件

    • shallowMount：浅渲染，只挂载一个组件而不渲染其子组件 (即保留它们的存根)

      • 在测试用例中，通常希望专注在一个孤立的单元中测试组件，避免对其子组件的行为进行间接的断言。额外的，对于包含许多子组件的组件来说，整个渲染树可能会非常大。重复渲染所有的子组件可能会让我们的测试变慢。但有时确实需要同时渲染好子组件，具体情况具体分析。

      • Wrapper：shallowMount和mount一样都返回一个包裹器Wrapper，通过这个包裹器我们可以访问Vue的实例，同时包裹器中也包含很多其他的属性和方法，两者不同的是被存根的子组件。mount是完整的渲染，直接连子组件一同渲染了，shallowMount需要我们再单独处理这个一层。

        js
        describe('HelloWorld.vue', () => {
  it('renders props.msg when passed', () => {
    const msg = 'new message'
    const wrapper1 = mount(HelloWorld, {
      props: { msg }
    })
    const wrapper2 = shallowMount(HelloWorld, {
      props: { msg }
    })
    console.log(wrapper1.html())
    console.log(wrapper2.html())
  })
})

        得到的结果：

        html
        console.log(wrapper1.html())的结果
<div class='hello'>
  <h1>new message</h1>
  <div>这是hello组件</div>
</div>

console.log(wrapper2.html())的结果
<div class='hello'>
  <h1>new message</h1>
  <hello-stub></hello-stub>
</div>

    • creatLocalVue：创建独立作用域，一般是为了配合测试Vuex使用，作用避免污染 Vue 的基础构造函数。

      js
      import Vuex from 'vuex'
import { shallowMount, createLocalVue } from '@vue/test-utils';
import StackManage from '@/views/DataStack/components/StackManage/index.vue'

// 创建独立作用域Vue，避免影响全局
const localVue = createLocalVue()
localVue.use(Vuex)

const store = new Vuex.Store({ storeConfig })

// 一个测试套件
describe('测试XXX组件', () => {
  it('组件成功挂载', async () => {
    const wrapperStackManage = shallowMount(StackManage, {
      store,
      localVue,
    })
    expect(wrapperStackManage.exists()).toBe(true)
  })
})

单元测试用例设计 ​

基本原则 ​

  • 测试代码时，只考虑测试，不考虑内部实现

  • 数据尽量模拟现实，越靠近现实越好

  • 充分考虑数据的边界条件

  • 对重点、复杂、核心代码，重点测试

  • 单元测试代码要尽量简单，如果测试代码太复杂，那么测试代码本身就可能有bug。利用AOP(beforeEach、afterEach)，减少测试代码数量，避免无用功能。

测试类型 ​

  • TDD(测试驱动开发)

    • 先写测试用例，再写功能模块。TDD的目的是通过测试用例来指引实际的功能开发，让开发人员首先站在全局的视角来看待需求。
    • 在目前互联网的开发环境下，业务开发很难做到TDD开发，一是因为需要更多时间编写单元测试用例；二是要求非常了解业务需求；三是要求开发人员有很强的代码设计能力。但是当我们写组件、工具方法、类库的时候，TDD就可以得到很好地使用。

  • BDD(行为驱动开发)

    • 先写主功能模块，再写测试用例。
    • 行为驱动开发要求更多人员参与到软件的开发中来，鼓励开发者、QA、相关业务人员相互协作。BDD是由商业价值来驱动，通过用户接口(例如GUI)理解应用程序。

  • 最后选择那种类型依据具体情况来看，可以先从BDD开始，目标是TDD形式。

常见用例 ​

  • 测试组件渲染输出

    • 是否成功挂载、文本是否正确、HTML结构是否正确、DOM属性和Class和Style是否是预期的、props是否一致，

    • 测试代码示例

      js
      it('组件成功挂载', () => {
  expect(wrapperStackList.exists()).toBe(true)
})
it('渲染成功应该包含数据源文本', () => {
  expect(wrapperStackList.text()).toContain('数据源')
})
it('应当有.StackListWrapper样式, 没有style样式', () => {
  const dom = wrapperStackList.find('div')
  // 测试是否是.StackListWrapper样式, 是否有style
  expect(dom.classes()).toContain('StackListWrapper')
  expect(dom.attributes().style).toBeFalsy()
})

  • 测试组件方法

    • 测试组件方法常见的就是测试其内部自有方法，借助warpper中的vm实例调用该方法并进行断言。

    • 需要注意的是，任何导致操作 DOM 的改变都应该在断言之前 await nextTick 函数。因为 Vue 会对未生效的 DOM 进行批量异步更新，避免因数据反复变化而导致不必要的渲染。在更新响应式 property 之后，我们可以直接 await 类似 trigger 或 trigger.vm.$nextTick 方法，等待 Vue 完成 DOM 更新。

    • 测试代码示例

      js
      describe('测试stackManage组件中的tabChange方法', () => {
  it('测试页签切换为数据管理时,组件文本内容应当包含"新增数据"同时不包含"新增列"', async () => {
    const wrapperStackManage = shallowMount(StackManage, {
      store,
      router,
      localVue,
    })
    // 更改data中的属性
    wrapperStackManage.setData({
      currentTabPane: 'column'
    })
    // 等待 Vue 完成 DOM 更新再进行断言判断
    await wrapperStackManage.vm.$nextTick()
    expect(wrapperStackManage.vm.currentTabPane).toMatch(/column/)
    expect(wrapperStackManage.text()).not.toContain('新增数据')
    expect(wrapperStackManage.text()).toContain('新增列')
    // 调用了组件方法
    wrapperStackManage.vm.tabChange('data')
    // 等待 Vue 完成 DOM 更新再进行断言判断
    await wrapperStackManage.vm.$nextTick()
    expect(wrapperStackManage.vm.currentTabPane).toMatch(/data/)
    expect(wrapperStackManage.text()).not.toContain('新增列')
    expect(wrapperStackManage.text()).toContain('新增数据')
  })
})

  • 测试自定义事件

js
  // 可以通过访问子组件实例来触发一个自定义事件。在`Vue-Test-Utils`中，一个组件它发射的事件可以通过`wrapper.emitted()`获取，它的返回值是一个对象，其中事件名作为对象的键名，对应的参数作为键的值
  it('测试emit触发cancel事件', () => {
  	wrapperTableInfoModal.vm.$emit('cancel')
  	const emitted = wrapperTableInfoModal.emitted()
  	expect(emitted.cancel).toBeTruthy()
  })

  • 测试Vuex

    • 单独测试store中的每一个部分：可以把store中的mutations、actions和getters单独划分，分别进行测试。

      • 单独测试store中的每一部分的好处是：单元测试可以小而且聚焦，当一个单元测试用例失败时，能够十分确切的知道错在哪里。缺点是：经常需要模拟Vuex的某些功能，而越多的模拟意味着越偏离实际，有时候很可能模拟错误而引入bug

    • 组合测试store：不拆分store，而是把它当做一个整体，测试store实例，进而希望它能按期望输出。

      • 组合测试store的好处是：这种方法更加健壮，因为不需要在重新编写、模拟Vuex的功能。

      • 组合测试store一个需要注意的点是，需要使用localVue而不是全局Vue上挂载Vuex

      • 组合测试代码例：

        js
        describe('测试Vuex Store', () => {
  it('测试set_dataStackList更新state.dataStackList的值为{ num: 1 }', () => {
    const store = new Vuex.Store({
      modules: {
        datastack: dataStackStoreConfig
      }
    })
    expect(store.state.datastack.dataStackList).toEqual([])
    const data = [{ num: 1 }]
    store.commit('datastack/SET_DATASTACK_LIST', data)
    expect(store.state.datastack.dataStackList).toEqual(data)
  })
})

  • 模拟用户交互

    • 可以借助trigger触发类似点击事件来模拟用户交互从而进行测试。

    • trigger：在该 wrapper DOM 节点上异步触发一个事件。trigger 带有一个可选的 options 对象。options 对象内的属性会被添加到事件上。trigger 会返回一个 Promise，当这个 Promise 被解决时，会确保组件已经被更新。

  const tabButDiv = wrapperStackList.find('.StackList-header_btn')
  const tabBut = tabButDiv.find('button')
  tabBut.trigger('click').then(() => {
    expect(wrapperStackList.vm.tableInfoModal).toBeTruthy()
  })
})

  • 测试异步代码

    异步代码在JavaScript中很常见，针对异步代码通常有两种场景：

    • 回调函数

      • 回调函数是非常常见的异步代码，比如像下面的picCompress方法，获取base64字符串数据后就会调用 callback(base64)。可以向测试用例的函数新增一个 done 参数后，断言给回调传入的参数是标准格式的base64字符串，jest 会等待 done 被调用，然后再结束本次测试。

      • 待测试的方法

        js
        /**
 * 图片压缩 借助canvas对图片进行重绘（canvas2DataURL）
 * @param {String} base64Url base64格式的图片字符串
 * @param {Object} imgAttr  重绘图片宽度、高度、质量等属性的设置
 * @param {Function} callback 该函数执行完毕后，要执行的回调函数，传入的参数即经过压缩后的图片base64值
 */
function picCompress(base64Url, imgAttr, callback) {
  const img = new Image();
  img.src = base64Url;
  img.onload = function () {
    const that = this;
    // 默认按比例压缩
    let w = that.width;
    let h = that.height;
    // 宽高比例
    const scale = w / h;
    w = imgAttr.width || w;
    h = imgAttr.height || (w / scale);
    // 默认图片质量为0.7
    let qualityf = 0.7;
    // 生成canvas
    const canvas = document.createElement('canvas');
    const ctx = canvas.getContext('2d');
    // 创建属性节点
    const anw = document.createAttribute('width');
    anw.nodeValue = w;
    canvas.setAttributeNode(anw);
    const anh = document.createAttribute('height');
    anh.nodeValue = h;
    canvas.setAttributeNode(anh);
    ctx.drawImage(that, 0, 0, w, h);
    // 图像质量（0-1）
    if (imgAttr.quality && imgAttr.quality <= 1 && imgAttr.quality > 0) {
      qualityf = imgAttr.quality;
    }
    // 重绘后的图片类型
    const imgType = imgAttr.type || 'image/jpeg';
    // quality值越小，所绘制出的图像越模糊
    const base64 = canvas.toDataURL(imgType, qualityf);
    // 回调函数返回base64的值
    callback(base64);
  }
}

      • 测试代码示例

        js
        it('测试picCompress方法中传入callback中的参数应当是base64字符串', (done) => {
    const baseURL = 'data:image/png;base64,iVBORw0KG...'
    const imgAttr = {
      quality: 0.7
    }

      try {
        expect(data).toMatch(base64Reg)

      }

  })

    • Promise

      • 还有常见的场景就是Promise，可以有两种方式测试，参看下方测试代码示例。

      • 待测试的方法

        js
        /**
 * 读取本地图片文件转换成base64数据在页面显示
 * @param {String} picPath 图片文件的路径
 */
function getBase64(picPath) {
  return new Promise((resolve) => {
    if (picPath) {
      const suffix = picPath.split('.')[1]
      fs.readFile(picPath, (err, data) => {
        if (err) throw err;
        const imageBase64 = data.toString('base64');
        const imagePrefix = `data:image/${suffix};base64,`;
        const image = imagePrefix + imageBase64
        resolve(image)
      });
    }
  });
}

      • 测试代码示例

        js
        it('测试getBase64方法应当返回base64字符串', async () => {
    const picPath = './tests/assets/studioIcon.png' -> base64
    // 方式一：可以直接在 test 中返回一个 Promise 对象，Jest 会等待该 Promise 被处理，如果 Promise 被拒绝，测试将自动失败。一定要记得返回 promise - 如果忘记编写 `return` 语句，测试会提前结束，导致无法执行 `.then()` 中的回调
    // return getBase64(picPath).then((data) => {
    //   expect(data).toMatch(base64Reg);
    // })
    // 方式二：在测试中使用 `async` 和 `await`。在 `test` 传入测试函数时，增加 `async` 关键词，将其设为异步函数

具体事例分析 ​

  • 以测试工具函数方法为例，前提尽量保证这个函数不能包含太多功能，拆分成一个个功能函数最佳，因为这样测试起来会很方便。以getBase64为例：

    js
    /**
 * 读取本地图片文件转换成base64数据在页面显示
 * @param {String} picPath 图片文件的路径
 */
function getBase64(picPath) {
  return new Promise((resolve) => {
    if (picPath) {
      const suffix = picPath.split('.')[1]
      fs.readFile(picPath, (err, data) => {
        if (err) throw err;
        const imageBase64 = data.toString('base64');
        const imagePrefix = `data:image/${suffix};base64,`;
        const image = imagePrefix + imageBase64
        resolve(image)
      });
    }
  });
}

  • 分析需求逐条罗列

    • 将分析出来的需求一条条罗列下来，之后挨个针对单个需求设计测试用例
      • 输入：图片路径字符串
      • 输出：该图片的base64字符串
        • 输入正确路径字符串 - 输出结果应当是符合base64规范字符串
        • 输入错误参数 - 应当抛出一个错误

  • 测试脚本编写

    • 逻辑不用太复杂，尽量避免测试代码过于复杂产生新bug，最常用的就是一个一个断言函数。

      • 输入正确图片路径字符串，应当返回符合base64规范的字符串

        js
        it('测试输入正确图片路径字符串,getBase64方法应当返回base64字符串', async () => {
  const picPath = './tests/assets/studioIcon.png'
  const data = await getBase64(picPath)
  expect(data).toMatch(base64Reg);
})

      • 输入一个不存在的图片路径字符串应当抛出一个错误

        js
        it('测试输入一个不存在的图片路径字符串,getBase64方法应当抛出一个错误', () => {
  const errPicPath = './tests/assets/studioIcon1.png'
  expect(getBase64(errPicPath)).rejects.toThrow()
})

Tips ​

  • tests目录中的各测试文件的目录规范建议可以就按照项目的目录规范来。
  • 编写测试代码时Jest在终端无法输出console日志，此时可以观察Jest-Electron弹出窗口的控制台日志。
  • 单元测试通过了并不意味着程序就没有bug了，但是不通过程序肯定有bug。
