On this page

在 Electron 和 Tauri 双框架共存的项目中，若需要遵循**开闭原则（OCP）**扩展第三种框架（如 Capacitor、Flutter Desktop 等），核心思路是：通过抽象隔离框架差异，使新增框架时无需修改现有代码。以下是具体架构设计：

开闭原则（Open-Closed Principle, OCP）是面向对象设计的五大SOLID原则之一，由 Bertrand Meyer 提出。它的核心定义是：

软件实体（类、模块、函数等）应该对扩展开放，但对修改关闭。 （Open for extension, closed for modification.）

核心理解

  1. 对扩展开放（Open for extension）
    • 当需求变化时，可以通过添加新的代码来扩展功能，而不是修改已有的代码。
    • 例如：通过继承、接口、组合等方式扩展行为。
  2. 对修改关闭（Closed for modification）
    • 已有的、经过测试的代码应尽量保持稳定，避免直接修改其内部逻辑，以减少引入新错误的风险。

1. 抽象核心接口（对扩展开放） ​

定义统一的抽象层，将框架相关的操作（如窗口管理、文件系统、IPC通信等）抽象为接口，具体框架实现这些接口。

typescript

js
// 核心抽象层（所有框架需实现此接口）
interface DesktopFramework {
  createWindow(config: WindowConfig): Promise<WindowHandle>;
  readFile(path: string): Promise<Uint8Array>;
  onEvent(event: string, callback: (data: any) => void): void;
  // ...其他通用能力
}

2. 实现具体框架适配器（闭修改） ​

为每个框架编写独立的适配器，实现核心接口。新增框架时只需添加新适配器，不修改旧代码。

typescript

js
// Electron适配器
class ElectronFramework implements DesktopFramework {
  async createWindow(config: WindowConfig) { /* Electron具体实现 */ }
  // ...其他方法
}

// Tauri适配器
class TauriFramework implements DesktopFramework {
  async createWindow(config: WindowConfig) { /* Tauri具体实现 */ }
  // ...其他方法
}

// 新增的第三种框架适配器（如Capacitor）
class CapacitorFramework implements DesktopFramework {
  async createWindow(config: WindowConfig) { /* Capacitor具体实现 */ }
  // ...其他方法
}

3. 依赖注入与动态选择（解耦） ​

通过工厂模式或环境变量动态选择框架，避免硬编码依赖。

typescript

js
// 框架工厂（根据配置返回具体实例）
function createFramework(): DesktopFramework {
  const frameworkType = process.env.FRAMEWORK; // 如 'electron' | 'tauri' | 'capacitor'
  
  switch (frameworkType) {
    case 'electron': return new ElectronFramework();
    case 'tauri': return new TauriFramework();
    case 'capacitor': return new CapacitorFramework(); // 扩展时仅需新增case
    default: throw new Error('Unsupported framework');
  }
}

// 业务代码仅依赖抽象
const framework = createFramework();
await framework.createWindow({ title: 'App' });

4. 统一构建与配置 ​

  • 构建工具：使用条件编译（如Webpack的externals或Rollup的output.globals）分离不同框架的打包逻辑。

  • 配置文件：通过JSON或环境变量声明当前使用的框架，例如：

    json
    { "framework": "capacitor" }

5. 框架特有逻辑的隔离 ​

  • 分支处理：若某些框架需要特殊逻辑，通过抽象接口的扩展方法实现，而非直接修改原有适配器。

    typescript
    interface ExtendedFramework extends DesktopFramework {
  frameworkSpecificMethod?(): void; // 可选扩展
}

// 使用时检查
if ('frameworkSpecificMethod' in framework) {
  framework.frameworkSpecificMethod();
}

6. 测试与扩展验证 ​

  • Mock测试：基于抽象接口编写单元测试，确保新增框架适配器符合预期行为。
  • 新增框架步骤：
    1. 实现新的DesktopFramework适配器。
    2. 在工厂中注册新框架。
    3. 更新构建配置（如新增capacitor的打包脚本）。

架构优势 ​

  • 符合OCP：扩展新框架时只需新增代码（适配器+工厂注册），无需修改Electron/Tauri的现有逻辑。
  • 低耦合：业务代码仅依赖抽象接口，与具体框架无关。
  • 易于维护：框架差异集中管理，升级或替换框架时影响范围可控。

│   └── FrameworkFactory.ts      # 动态创建实例
