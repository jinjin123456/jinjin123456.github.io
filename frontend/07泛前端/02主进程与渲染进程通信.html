核心概念 ​

主进程 ​

  • 每个Electron应用都有一个单一的主进程，作为应用程序的入口点。主进程运行在Node.js环境，因此可以使用Node.js的所有API能力。主进程并不在浏览器环境中运行，因此不进行页面渲染。

渲染进程 ​

  • 渲染进程说白了就是平时的Web页面前端开发。每个Electron应用都会为每个打开的BrowserWindow生成一个单独的渲染进程。但是渲染进程是无法直接调用Node.js的API的。

配置渲染进程Node能力 ​

方式一：Node intergration ​

  • 原理

  • 在每次new BrowserWindow创建新窗口的时候配置Node intergration字段

    js
    const mainWindow = new BrowserWindow({
    webPreferences: {
      nodeIntegration: true, // 是否启用Node intergration，即是否允许渲染进程使用node
      contextIsolation: false, // 不允许自定义preload脚本
    }
  })

  • 渲染进程中直接使用Node和Electron能力

    js
    import { getCurrentWindow } from 'electron';

getCurrentWindow().close() // 渲染进程直接调用electron能力关闭窗口

  • 小结

    • 优点：可以在渲染进程直接使用Node.js和electron能力
    • 缺点：安全性降低，并且渲染进程不能单独调试，无法避免避免不安全的JavaScript代码随意使用node环境的问题

方式二：preload脚本 ​

  • preload的工作是在创建窗体时预加载需要的node模块到渲染进程，然后以API方式暴露给渲染进程调用，preload共享渲染进程的window、document对象，因此preload可以轻松操作DOM，而渲染进程不共享preload的global对象。

  • preload的方式是在打开窗口前先预加载一段脚本，不需要修改webPreferences中的nodeIntegration和contextIsolation属性，主进程中，打开窗口时添加preload脚本

    js
    const mainWindow = new BrowserWindow({
    webPreferences: {
      preload: join(__dirname, '../preload/index.js'),
      sandbox: false
    }
  })
    js
    // src/preload/index.ts
import { contextBridge } from 'electron'
import { electronAPI } from '@electron-toolkit/preload'
const fs = require('fs'); 

// 这里就可以自定义需要暴露给渲染进程的一些api
 const api = {
   writeFile: (filename, text, callback) => {
     fs.writeFile(filename, text, callback);
   }
 }

// Use `contextBridge` APIs to expose Electron APIs to
// renderer only if context isolation is enabled, otherwise
// just add to the DOM global.
if (process.contextIsolated) {
  try {
    contextBridge.exposeInMainWorld('electron', electronAPI)
    contextBridge.exposeInMainWorld('api', api)
  } catch (error) {
    console.error(error)
  }
} else {
  window.electron = electronAPI
  window.api = api
}

  • 渲染进程中，通过window.xxx适用preload脚本注入的能力

    js
    const { api: { writeFile } } = window

writeFile('1.txt', 'txt', (err) => {
    if (!err) console.log('完成写入')
})

  • 小结

    • 优点：可以控制暴露哪些api，安全性提高，而且渲染进程可以单独调试部分功能
    • 缺点：相应的较麻烦的是所有渲染进程想使用的能力，都需要通过preload脚本注入

推荐 ​

  • 实际开发中还是推荐选择预加载脚本这种方式配置渲染进程的Node或Electron能力。

进程间通信的基本模式 ​

  • 在 Electron 中，进程使用 ipcMain 和 ipcRenderer 模块，通过开发人员定义的“通道”传递消息来进行通信，即IPC通道。

模式1：渲染进程到主进程(单向) ​

  • 渲染进程通过 ipcRenderer.send API 发送消息，然后主进程通过 ipcMain.on API 接收。适用于从web渲染进程中需调用主进程能力的场景，比如点击渲染进程按钮最小化整个应用窗口

    js
    // 渲染进程页面文件
oBtn.addEventListener('click', () => {
  ipcRenderer.send('window-min')
})
    js
    // 主进程文件
ipcMain.on('window-min', () => {
   !mainWindow.isMinimized() && mainWindow.minimize()
})

模式2：渲染进程到主进程(双向） ​

  • 双向IPC 的一个常见应用是从渲染器进程代码调用主进程模块并等待结果。可以通过将 ipcRenderer.invoke 与 ipcMain.handle 搭配使用来完成。比如等待文件对话框返回文件路径

    js
    // 渲染进程页面文件
oBtn.addEventListener('click', async () => {
    const filePath = await ipcRenderer.invoke('dialog:openFile')
    ElMessage.info(filePath)
})
    js
    // 主进程文件
async function handleFileOpen() {
  const { canceled, filePaths } = await dialog.showOpenDialog()
  if (!canceled) {
    return filePaths[0]
  }
}
ipcMain.handle('dialog:openFile', handleFileOpen)

模式3：主进程到渲染进程 ​

  • 将消息从主进程发送到渲染器进程时，需要指定是哪一个渲染器接收消息。 消息需要通过其 WebContents 实例发送到渲染器进程。 此 WebContents 实例包含一个 send 方法，其使用方式与 ipcRenderer.send 相同。

    js
    // 渲染进程页面文件
const handleShowClick = (e, msg) => { // 必须要显式接收一下e参数
    ElMessage.info(msg)
}
onMounted(() => {
    ipcRenderer.removeListener('show-click', handleShowClick) // 避免重复监听
    ipcRenderer.on('show-click', handleShowClick)
})
    js
    // 主进程文件
// 在托盘菜单中每点击一次显示就弹出信息提示
const trayTemplate = [
    {
        label: '显示',
        click: () => {
            if (!mainWindow) return
            // if (mainWindow.isMinimized()) mainWindow.restore()
            if (mainWindow.isMinimized()) {
                mainWindow.restore()
                // 注意这里虽然只传了一个参数，但接收的时候是两个参数
                mainWindow.webContents.send('show-click', '用户点击了显示')
            }
        }
    },
    ...
]
const contextMenu = Menu.buildFromTemplate(trayTemplate)
const tray = new Tray(trayIcon)
tray.setContextMenu(contextMenu)
...

模式4：渲染进程与渲染进程之间通信 ​

  • 没有直接的方法可以使用 ipcMain 和 ipcRenderer 模块在渲染进程之间发送消息。 有两个方案：

    • 方案1：将主进程作为“中转站”，将消息从一个渲染进程发送到主进程，然后主进程将消息转发到另一个渲染进程，相当于模式1+模式3。

    • 方案2：从主进程将一个 MessagePort 传递到两个渲染器。MessagePort是一个允许在不同上下文之间传递消息的Web功能。 就像 window.postMessage, 但是在不同的通道上。基本思路如下方代码所示：

      js
      // 主进程文件
/*
主进程设置了一个MessageChannel，然后将每个端口发送给不同的渲染进程。 这样可以让渲染进程彼此之间发送消息，而无需使用主进程作为中转。
*/
const { BrowserWindow, app, MessageChannelMain } = require('electron')

app.whenReady().then(async () => {
  // 创建窗口
  const mainWindow = new BrowserWindow({
    show: false,
    webPreferences: {
      contextIsolation: false,
      preload: 'preloadMain.js'
    }
  })

  const secondaryWindow = new BrowserWindow({
    show: false,
    webPreferences: {
      contextIsolation: false,
      preload: 'preloadSecondary.js'
    }
  })

  // 建立通道
  const { port1, port2 } = new MessageChannelMain()

  // webContents准备就绪后，使用postMessage向每个webContents发送一个端口。
  mainWindow.once('ready-to-show', () => {
    mainWindow.webContents.postMessage('port', null, [port1])
  })

  secondaryWindow.once('ready-to-show', () => {
    secondaryWindow.webContents.postMessage('port', null, [port2])
  })
})
      js
      // 预加载脚本 preloadMain.js and preloadSecondary.js (Preload scripts)
/*
在你的预加载脚本中通过IPC接收端口，并设置相应的监听器
*/
const { ipcRenderer } = require('electron')

ipcRenderer.on('port', e => {
  // 接收到端口，使其全局可用。
  window.electronMessagePort = e.ports[0]

  window.electronMessagePort.onmessage = messageEvent => {
    // 处理消息
      console.log(messageEvent)
  }
})
      js
      // 渲染进程文件
/*
window.electronMessagePort 在全局范围内可用，你可以在应用程序的任何地方调用postMessage 方法，以便向另一个渲染进程发送消息
*/
window.electronMessagePort.postMessage('ping')
